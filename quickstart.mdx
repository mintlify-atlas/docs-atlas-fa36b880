## Prerequisites

- A running Java application you want to profile
- async-profiler installed — see [Installation](/installation)
- On Linux: `kernel.perf_event_paranoid=1` and `kernel.kptr_restrict=0` set (see [Installation](/installation#configure-linux-kernel-parameters))

## Profile your application

<Steps>
  <Step title="Find the process ID">
    Use `jps` to list running Java processes:

    ```bash
    jps
    ```

    Example output:

    ```
    9234 Jps
    8983 Computey
    ```

    Alternatively, use `pgrep` to search by name:

    ```bash
    pgrep -a java
    ```

    Note the PID of the application you want to profile — `8983` in the example above.
  </Step>
  <Step title="Run the profiler">
    Profile the application for 30 seconds and save the result as an interactive flamegraph:

    ```bash
    asprof -d 30 -f flamegraph.html <PID>
    ```

    Replace `<PID>` with the process ID from the previous step.

    <Tip>
      If only one Java process is running on the system, you can use the `jps` keyword instead of a numeric PID:

      ```bash
      asprof -d 30 -f flamegraph.html jps
      ```

      You can also use the application name as it appears in `jps` output (for example, `Computey`).
    </Tip>
  </Step>
  <Step title="View the flamegraph">
    Open `flamegraph.html` in any browser. The flamegraph shows the call stacks sampled during profiling — wider frames indicate more CPU time spent in that method or its callees.

    To save results in text format instead, omit `-f` or use a `.txt` extension:

    ```bash
    asprof -d 30 <PID>
    ```

    Text output looks like this:

    ```
    --- Execution profile ---
    Total samples:           687
    Unknown (native):        1 (0.15%)

    --- 6790000000 (98.84%) ns, 679 samples
      [ 0] Primes.isPrime
      [ 1] Primes.primesThread
      [ 2] Primes.access$000
      [ 3] Primes$1.run
      [ 4] java.lang.Thread.run

              ns  percent  samples  top
      ----------  -------  -------  ---
      6790000000   98.84%      679  Primes.isPrime
        40000000    0.58%        4  __do_softirq
    ```

    This output shows that `Primes.isPrime` is the hottest method, consuming 98.84% of sampled CPU time, called from the `Primes.primesThread` call stack. The `__do_softirq` frame is a kernel frame — visible because async-profiler captures the full stack including native and kernel frames.
  </Step>
  <Step title="Start and stop separately (optional)">
    For more control, you can attach the profiler and detach it independently instead of using a fixed duration:

    ```bash
    asprof start <PID>
    # ... exercise your performance scenario ...
    asprof stop <PID>
    ```

    Output and profiling results are printed to the console where you ran `asprof stop`.
  </Step>
</Steps>

## Common options

| Option | Description |
|---|---|
| `-d <seconds>` | Profile for a fixed duration |
| `-f <file>` | Save output to a file (`.html` for flamegraph, `.jfr` for JFR) |
| `-e <event>` | Profiling event: `cpu` (default), `alloc`, `lock`, `wall`, etc. |
| `-i <interval>` | Sampling interval (default: `10ms` for CPU profiling) |
| `--title <text>` | Set the flamegraph title |
